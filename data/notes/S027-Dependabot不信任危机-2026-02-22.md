# S027 - Dependabot 不信任危机

> 开发者主动关闭 Dependabot："它带来的麻烦比解决的问题多"

---

## 核心观察

Go 语言核心贡献者 Filippo Valsorda 发布文章《Turn Dependabot Off》，引发开发者广泛共鸣。核心论点：Dependabot 的频繁 PR 干扰开发流程，创造"虚假安全感"，带来的维护负担超过安全收益。Lobsters 92 分讨论显示这不是个人意见。

## 痛点细节

### 1. 信号噪音问题

- Dependabot 每周/每月创建大量 PR
- 大多数是"虚假警报"或无关紧要的版本更新
- 开发者形成"Dependabot PR 盲视忽略"

> "Dependabot trains you to ignore its PRs. That's the opposite of what a security tool should do." — Filippo Valsorda

### 2. 虚假安全感

| 假设 | 现实 |
|------|------|
| Dependabot = 安全 | Dependabot 只更新直接依赖 |
| 有 PR 就看 | 大多数 PR 被自动合并或忽略 |
| 版本新 = 更安全 | 新版本可能引入新 bug |

### 3. 维护负担

- CI 时间增加（每个 Dependabot PR 都跑 CI）
- 合并冲突频繁
- "它创造的工作量超过它节省的"

### 4. 核心洞察

> "Security is about understanding your attack surface, not about blindly bumping versions."

Dependabot 的设计假设：**更新 = 更安全**。但真正需要的是：**理解攻击面**。

## 潜在产品方向

### 🎯 高潜力

1. **智能依赖更新工具**
   - 只在"真正安全相关"时发送 PR
   - 分析 CVE 影响范围，而非盲目更新
   - 产品方向：Dependabot 的"安静模式"

2. **攻击面可视化平台**
   - 显示哪些依赖真正暴露在攻击面
   - 优先级排序安全更新
   - "Focus on what matters"

3. **依赖安全审计工具**
   - 不自动更新，但提供清晰的安全报告
   - 周报而非每日 PR
   - 可配置的"打扰频率"

### 💡 中潜力

4. **Dependabot 配置优化服务**
   - 帮助团队配置 Dependabot 规则
   - 减少噪音，保留信号
   - 咨询 + 工具

5. **依赖更新中心化管理**
   - 团队统一审核依赖更新
   - 避免"每个人都收到 PR"
   - 企业级依赖策略

## 更广泛的信号：自动化工具信任危机

Dependabot 不是唯一被质疑的自动化工具：

| 工具 | 质疑点 |
|------|--------|
| Dependabot | 噪音太多，虚假安全感 |
| Copilot | 代码质量不可控 |
| CodeQL | 误报太多 |
| Renovate | 类似 Dependabot |

**模式**：开发者开始质疑"自动化 = 好"的假设。
