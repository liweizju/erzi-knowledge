## 30 秒结论

AI Agent 开发正从"prompt engineering"转向"context engineering"。核心认知：LLM 有有限的"注意力预算"，上下文越长效果越差（context rot）。开发者需要精细化上下文管理，而非简单堆砌 tokens。产品机会：Context 管理工具、Agent 开发框架、上下文优化服务。

---

## 核心认知

### Prompt Engineering vs Context Engineering

| 维度 | Prompt Engineering | Context Engineering |
|------|-------------------|-------------------|
| 焦点 | 写好系统提示词 | 管理整个上下文状态 |
| 时间范围 | 单次 inference | 多轮对话 + 长期运行 |
| 信息源 | 系统指令 | 系统指令 + 工具 + MCP + 外部数据 + 历史消息 |
| 优化目标 | 输出质量 | 上下文效用最大化 |

### Context Rot（上下文腐烂）

- LLM 在长上下文中信息检索准确率下降
- Transformer 架构导致 n² 注意力计算
- 模型训练数据中短序列更常见
- 上下文窗口不是"越大越好"，而是"越精越好"

### 注意力预算

```
上下文 tokens → 消耗注意力预算 → 边际效用递减
```

**Anthropic 的建议**：
> "找到最小的、高信号 tokens 集合，最大化目标结果概率"

## 技术细节

### 有效上下文的构成

1. **系统提示词**
   - 清晰、直接、正确"海拔"
   - 避免"过度工程"和"过于模糊"
   - 在具体和灵活之间平衡

2. **工具定义**
   - 只包含必要的工具
   - 工具描述精确但简洁

3. **MCP（Model Context Protocol）**
   - 外部数据源连接
   - 需要数据边界控制

4. **消息历史**
   - 迭代精简，不是无限保留
   - 保留高信号交互

5. **外部数据**
   - RAG 检索结果
   - 需要相关性过滤

### Context Engineering 循环

```
1. 评估当前上下文状态
2. 决定哪些信息进入下一轮
3. 精简/压缩/丢弃低价值信息
4. 传递给模型 inference
5. 返回步骤 1
```

## 建议动作

### Agent 开发者

1. **审计当前上下文使用**
   - 测量每个 token 的实际贡献
   - 识别"噪音"和"冗余"

2. **实现上下文精简机制**
   - 设置 token 上限（如 20,000）
   - 实现自动压缩/摘要
   - 建立信息优先级

3. **监控上下文健康**
   - 跟踪上下文长度趋势
   - 检测 context rot 症状（准确率下降）

### 产品团队

1. **评估工具必要性**
   - 每个工具是否真正需要？
   - 工具描述是否精确？

2. **设计上下文策略**
   - 不同场景使用不同上下文配置
   - 避免一刀切

---

## 风险 / 不确定性

1. **模型差异**
   - 不同模型上下文行为不同
   - Claude vs GPT vs Gemini 可能需要不同策略

2. **缺乏标准化工具**
   - Context engineering 仍在早期
   - 最佳实践尚未固化

3. **性能 vs 成本权衡**
   - 精简上下文需要额外计算
   - 可能增加延迟
